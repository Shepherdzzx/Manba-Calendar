# 任务

**题目一：语音版的日历工具**请开发一个语音日历工具，帮助用户提高日历管理的效率和便捷性。

要求：请了解用户在日历管理中的真实需求，设计并实现一个以语音交互为核心的日历管理工具，能准确、顺畅地实现通过语音添加/删除/查看事件提醒等能力。

# 要求

* 请基于 PR 添加新功能。
* 每个PR只做一件事：每个 PR只实现或修改单一功能；鼓励尽可能小、粒度尽可能细的 PR；大功能应拆分为多个独立 PR 分步提交。
* PR标题与描述需清晰完整，内容包含：

  * 标题：一句话说明本 PR 新增/修改了什么。
  * 功能描述：说明该功能的作用与使用方式。
  * 实现思路：简要说明技术选型或核心实现逻辑。
  * 测试方式：如何验证该功能正常运行。
* PR合并后，主分支代码需保持可运行状态，评委在任意时间查看应能复现演示效果。

# 对Flutter应用的要求

1. 应用配色可选,在设置中可选择不同配色,默认配色是紫金
2. 识别语音时是按住识别,松开后进行解析
3. 应用应简洁明了,首页即是日历,点击日期可查看该日期对应的事件,并配备增删改的选项.
4. 如果对插图,背景图有要求请尽可能描述,我会将描述发给文生图的模型.

# 大致方向

可以将项目定位为一个“面向高频日程记录场景的语音日历助手”，核心目标不是做完整办公套件，而是优先解决“说一句话就能快速记下、查看、调整日程”这件事。

建议的产品方向如下：

1. 先聚焦三个核心能力：

   - 语音添加事件：例如“明天下午三点和张三开会”。
   - 语音查看日程：例如“帮我看看今天还有什么安排”。
   - 语音删除或取消事件：例如“取消今晚七点的健身”。
2. 交互流程以“语音输入 + 文本确认”作为第一阶段方案：

   - 用户说出自然语言指令。
   - 系统先完成语音转文字。
   - 再从文字中提取时间、事件名称、提醒方式等结构化信息。
   - 最后用清晰的文本结果让用户确认，避免误识别直接写入日历。
3. 技术实现上建议拆成四层：

   - 语音输入层：负责录音、上传音频、触发识别。
   - 语音识别层：将中文语音转成文本。
   - 日程解析层：把“明天早上”“下周一下午三点”这类自然表达转换为标准时间，并识别动作类型（新增、查询、删除）。
   - 日历管理层：真正执行事件的新增、查询、删除和提醒。
4. 为了保证演示效果，第一版应优先支持常见中文时间表达：

   - 今天、明天、后天
   - 上午、下午、晚上
   - 几点、几点半
   - 周几、下周几
     先把高频表达做好，比一开始追求全量语义覆盖更稳妥。
5. 数据层可以先用本地存储或简单数据库保存事件：

   - 每条事件至少包含标题、时间、提醒时间、创建时间、状态。
   - 删除操作建议先做“软删除”或二次确认，减少语音误删风险。
6. 提醒能力可以作为后续增强：

   - 第一阶段先完成“添加 / 查看 / 删除”闭环。
   - 第二阶段再补充定时提醒、重复事件、修改事件等能力。
7. 如果计划接入 ASR 模型并使用 Flutter 开发 Android 应用，建议采用如下流程：

   - Flutter 负责 Android 客户端界面、录音按钮、权限申请、结果展示与确认交互。
   - 用户在 Flutter 端按下语音输入后，应用采集音频并发送给 ASR 模型。
   - ASR 模型先将语音内容转写成文本。
   - 程序将 ASR 输出文本与预先设计好的 prompt 一起发送给大模型。
   - prompt 需要明确要求模型只输出固定格式的 JSON，不返回解释、不返回多余文本。
   - 模型直接根据文本语义生成程序可执行的结构化指令，例如操作类型、事件标题、日期、时间、提醒状态等。
   - 例如可统一定义为：`{"intent":"create_event","title":"开会","date":"2026-05-30","time":"15:00","need_reminder":true}`。
   - 应用端对返回 JSON 做格式校验和字段校验，通过后再执行新增、查询、删除等日历操作。
   - 在真正写入数据前，Flutter 页面向用户展示解析结果并请求确认，以降低语音误识别或模型误解析带来的错误操作风险。
8. 这种方案的优点是实现路径更短：

   - ASR 模块解决“听清楚”。
   - Prompt + 大模型解决“理解语义并生成标准指令”。
   - Flutter 客户端解决“如何交互、展示结果和触发执行”。
     这样不需要单独开发一套复杂的语义解析器，更适合快速完成可演示版本。
9. 但为了保证稳定性，需要额外注意三点：

   - 必须严格约束模型输出格式，最好只允许返回固定字段的 JSON。
   - 程序侧不能直接信任模型输出，必须先校验字段是否完整、时间格式是否正确、intent 是否在允许范围内。
   - 对删除、覆盖、修改这类操作应增加确认步骤，避免错误执行。
10. PR拆分建议也可以围绕最小可演示单元推进：

- PR1：完成基础项目初始化和日历事件数据结构。
- PR2：完成事件的新增 / 查询 / 删除文本流程。
- PR3：完成 Flutter 端基础页面、录音入口与权限处理。
- PR4：接入 ASR 模型，实现语音转文字。
- PR5：接入 Prompt + 大模型输出 JSON 指令。
- PR6：增加 JSON 校验、执行确认与提醒能力优化。

整体上，这个项目最重要的是“语音输入自然、模型输出稳定、结构化结果可校验、关键操作可确认”，这样更符合真实用户需求，也更容易做出稳定、可展示的版本。
